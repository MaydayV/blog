## 🚀 快速开始

### 前置要求

- 一个 GitHub 账号
- 基本的 Git 操作知识
- Markdown 基础

### 快速投稿（5 分钟版）

```bash
# 1. Fork 仓库到您的账户
# 2. 克隆您的 fork
git clone https://github.com/YOUR_USERNAME/blog.git
cd blog

# 3. 创建文章
cp scaffolds/post.md source/_posts/$(date +%Y-%m-%d)-my-article.md

# 4. 编辑文章
# 使用您喜欢的编辑器修改文件

# 5. 提交并推送
git add .
git commit -m "Add: 文章标题"
git push origin main

# 6. 在 GitHub 上创建 Pull Request
```

## 📬 投稿流程

### 详细步骤

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   1. Fork   │───▶│   2. 编写   │───▶│   3. 提交   │
│    仓库     │    │    文章     │    │     PR      │
└─────────────┘    └─────────────┘    └──────┬──────┘
                                             │
                                             ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   6. 合并   │◀───│   5. 修改   │◀───│   4. 审核   │
│    发布     │    │   反馈      │    │     意见    │
└─────────────┘    └─────────────┘    └─────────────┘
```

#### 第一步：Fork 仓库

1. 访问 https://github.com/yourusername/blog
2. 点击右上角的 **Fork** 按钮
3. 等待 Fork 完成

#### 第二步：编写文章

1. 克隆您 Fork 的仓库
2. 在 `source/_posts/` 目录下创建文章
3. 遵循文章格式要求（见下文）
4. 如需添加图片，放入 `source/images/` 目录

#### 第三步：提交 Pull Request

1. 将更改推送到您的 Fork
2. 访问原仓库，点击 **New Pull Request**
3. 填写 PR 描述，说明文章内容
4. 提交 PR

#### 第四步：审核流程

1. 维护者会在 3-5 个工作日内审核
2. 如需修改，会在 PR 中留言
3. 修改后再次提交即可

#### 第五步：合并发布

审核通过后，文章将被合并并自动部署到博客。

## 📝 文章规范

### 文章类型

我们欢迎以下类型的文章：

✅ **技术教程**：编程语言、框架、工具使用指南
✅ **开发经验**：项目复盘、最佳实践、踩坑记录
✅ **技术思考**：行业观点、技术趋势、深度分析
✅ **开源分享**：开源项目介绍、贡献指南
✅ **工具推荐**：效率工具、开发神器介绍

### 禁止内容

❌ 政治敏感内容
❌ 色情、暴力、恐怖内容
❌ 歧视、仇恨言论
❌ 虚假信息、谣言
❌ 纯广告、营销内容
❌ 侵犯版权的内容

## 📐 格式要求

### 文件名格式

```
YYYY-MM-DD-article-title.md

示例：
✅ 2024-01-15-introduction-to-hexo.md
✅ 2024-02-20-vue3-composition-api-guide.md
❌ my-article.md           # 缺少日期
❌ 2024/01/15/article.md    # 使用子目录
```

### 文章头部 (Front-matter)

```markdown
---
title: "文章标题"
date: 2024-01-15 10:00:00
updated: 2024-01-16 15:30:00  # 可选
author: "作者名"              # 必填
tags:
  - 标签1
  - 标签2
categories:
  - 分类名
excerpt: "文章摘要，100-200字，用于列表页展示"
---
```

#### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | ✅ | 文章标题，建议使用引号包裹 |
| `date` | ✅ | 发布时间，格式 `YYYY-MM-DD HH:mm:ss` |
| `updated` | ❌ | 最后更新时间 |
| `author` | ✅ | 作者名字，将显示在文章底部 |
| `tags` | ❌ | 标签列表，建议 2-5 个 |
| `categories` | ❌ | 分类，单个文章建议 1-2 个分类 |
| `excerpt` | ❌ | 文章摘要，用于列表页展示 |

### 内容格式

#### 标题层级

```markdown
# 一级标题 - 文章标题（自动使用 front-matter 中的 title，正文中避免使用）

## 二级标题 - 主要章节

### 三级标题 - 小节

#### 四级标题 - 更细分的内容
```

#### 代码块

```markdown
使用三个反引号 + 语言标识

```javascript
function hello() {
  console.log("Hello, World!");
}
```

```bash
# 命令行示例
npm install hexo
```
```

#### 图片

```markdown
# 本地图片（推荐）
![图片描述](/images/your-image.png)

# 外部图片
![图片描述](https://example.com/image.png)
```

图片应放入 `source/images/` 目录，建议使用 WebP 格式，单张不超过 500KB。

#### 链接

```markdown
# 外部链接（自动在新标签页打开）
[链接文字](https://example.com)

# 内部链接
[另一篇文章](/2024/01/15/hello-world/)
```

#### 表格

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| A   | B   | C   |
| D   | E   | F   |
```

## ✅ 审核标准

### 内容质量要求

1. **原创性**：投稿文章必须为原创内容，或已获得原作者授权
2. **准确性**：技术内容需要准确无误，代码示例需要经过测试
3. **完整性**：文章应该有清晰的结构，包含引言、正文、总结
4. **可读性**：语言流畅，逻辑清晰，易于理解

### 格式要求

1. 符合上述格式规范
2. 无拼写和语法错误
3. 代码格式正确，有语法高亮
4. 图片清晰，有适当的尺寸

### 加分项

- 包含实际案例或项目经验
- 有配套的可运行代码仓库
- 使用图表、流程图辅助说明
- 提供进一步阅读的资源链接

## ❓ 常见问题

### Q1: 投稿后多久能收到回复？

通常在 3-5 个工作日内会收到审核反馈。如果超过一周没有回复，可以在 PR 中 @ 维护者询问进度。

### Q2: 一篇文章可以同时发布到其他平台吗？

可以。本博客接受非独家内容，您可以将文章同时发布到自己的博客、微信公众号、知乎专栏等平台。但请确保内容原创或已获得授权。

### Q3: 文章需要配图吗？

技术类文章建议包含配图，如：
- 架构图或流程图
- 关键代码截图
- 运行效果截图

如果没有图片也是可以的，但适当配图会让文章更生动。

### Q4: 投稿被拒绝的常见原因有哪些？

常见拒绝原因包括：

1. **内容质量问题**：
   - 内容过于简单，缺乏深度
   - 技术内容有误
   - 代码无法运行

2. **格式问题**：
   - 不符合格式规范
   - 缺少必要的 front-matter
   - 文件名格式错误

3. **内容违规**：
   - 包含禁止内容（广告、抄袭等）

4. **已有类似内容**：
   - 博客已有相同主题且内容质量相当的文章

### Q5: 如何修改已投稿的文章？

在 PR 被合并前，您可以随时修改：

```bash
# 在本地修改文件后
git add .
git commit --amend --no-edit  # 修改最后一次提交
git push origin main --force  # 强制推送到远程
```

PR 被合并后，需要提交新的 PR 来修改。

### Q6: 成为固定撰稿人有什么好处？

持续贡献优质内容的作者可以申请成为固定撰稿人，享受：

- 快速审核通道（1-2 个工作日）
- 作者专栏页面
- 优秀作者徽章
- 参与博客重大决策讨论

### Q7: 遇到技术问题怎么办？

如果在投稿过程中遇到技术问题：

1. 查阅 [Hexo 官方文档](https://hexo.io/docs/)
2. 在 PR 中描述问题，维护者会协助解决
3. 查看本仓库的 [Issues](https://github.com/yourusername/blog/issues) 是否有类似问题
